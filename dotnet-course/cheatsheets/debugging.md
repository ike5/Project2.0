# Debugging & Stack Traces Cheatsheet

The core skill of app support: turn an error into a root cause fast.

## Reading a stack trace

```
System.NullReferenceException: Object reference not set to an instance of an object.
   at TaskApi.Services.TaskService.Complete(Int32 id) in /src/Services/TaskService.cs:line 42
   at TaskApi.Controllers.TasksController.Complete(Int32 id) in /src/Controllers/TasksController.cs:line 28
   at lambda_method3(Closure, Object, Object[])
   ...
```

Read it **top-down**:
1. **Line 1** — the exception *type* and *message* (here: a `NullReferenceException`).
2. **First `at ...` with YOUR file/line** — where it was thrown:
   `TaskService.cs:line 42`. Start here.
3. **Subsequent frames** — the call chain that led there (controller → service).
4. Frames without your files (framework/lambda) are usually noise — find the
   topmost frame in *your* code.

> The **innermost** exception matters most. For wrapped errors, read the
> `---> InnerException` section; that's the real cause.

## Common .NET exceptions and what they usually mean

| Exception | Typical cause | First check |
|-----------|--------------|-------------|
| `NullReferenceException` | used something that was `null` | the variable on the throwing line; enable NRT warnings |
| `ArgumentNullException` / `ArgumentException` | bad arg passed to a method | the caller's argument |
| `InvalidOperationException` | object in wrong state (e.g. `Single()` found 0/2, modified collection while iterating) | the operation's preconditions |
| `KeyNotFoundException` | `dict[key]` for a missing key | use `TryGetValue` |
| `IndexOutOfRangeException` | bad array/list index | the index math |
| `FormatException` | `int.Parse` on non-numeric | use `int.TryParse` |
| `TaskCanceledException` / `TimeoutException` | a call took too long / token cancelled | the downstream dependency |
| `DbUpdateException` (EF Core) | constraint violation, bad migration | inner exception → SQL error |
| `HttpRequestException` | downstream HTTP failure | the URL, status, network |
| `FileNotFoundException` (assembly) | missing/mismatched dependency | deployed files, versions |

## Using the debugger (VS Code C# Dev Kit / Rider)

- **Breakpoint** — click the gutter (or `F9`). Execution pauses there.
- **Step**: `F10` step over · `F11` step into · `Shift+F11` step out · `F5` continue.
- **Watch / Locals** — inspect variable values at the pause.
- **Call Stack** — see how you got here; click frames to move up/down.
- **Conditional breakpoint** — right-click a breakpoint → break only when e.g. `id == 7`.
- **Exception breakpoints** — break the moment an exception is *thrown* (before it's
  caught/lost) — invaluable for "swallowed" errors.

`launch.json` (VS Code) defines how to start/attach the debugger; `dotnet` projects
get one generated by the C# Dev Kit.

## Diagnose-without-a-debugger (production)

```csharp
 logger.LogError(ex, "Failed to complete task {TaskId}", id);  // structured: includes the exception + data
```
- Read **logs** first (Module 06): find the exception + the structured fields around it.
- Reproduce locally with the same **inputs/config** from the logs.
- `dotnet-counters`, `dotnet-trace`, `dotnet-dump` are CLI diagnostics tools for live
  processes (perf, hangs, memory) — know they exist.

## The support reflex

1. **What's the exception type + message?** (top of the trace)
2. **Which line of *our* code threw it?** (first own-file frame)
3. **What was null/invalid there, and where did it come from?** (walk the stack down)
4. **What inputs/config triggered it?** (logs)
5. **Write a failing test that reproduces it**, then fix (Module 05).
